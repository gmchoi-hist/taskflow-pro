## 읽는 순서

```
00-overview.md → 01-product.md → 02-specs.md → 03-design.md → 04-tasks.md → 05-conventions.md
```

무엇(제품)을 먼저 이해한 뒤, 그것이 어떻게 동작해야 하는지(명세), 그것을 어떤 구조로 만들지(설계), 무엇부터 만들지(작업), 마지막으로 어떤 규칙을 지키며 만들지(컨벤션) 순서로 읽는다.

## 왜 이렇게 나누는가 — 관심사 분리(Separation of Concerns)

각 문서는 서로 다른 질문에 답하고, 서로 다른 속도로 변한다.

- **product**: "왜, 누구를 위해"를 다룬다. 가장 느리게 변한다.
- **specs**: "무엇을, 어떻게 동작해야 하는지"를 다룬다. 기능 추가/변경 시 갱신된다.
- **design**: "어떤 구조로 구현하는지"를 다룬다. 기술적 결정이 바뀔 때 갱신된다.
- **tasks**: "지금 무엇을 하고 있는지"를 다룬다. 가장 자주, 실시간으로 변한다.
- **conventions**: "어떻게 작성해야 하는지"를 다룬다. 팀/프로젝트 전반에 걸쳐 안정적으로 유지된다.

하나의 문서에 이 모든 것을 섞으면, 작은 변경(예: 작업 상태 업데이트)이 안정적인 내용(예: 제품 정의)과 뒤섞여 관리가 어려워지고, 필요한 정보를 찾기 위해 전체 문서를 훑어야 한다. 관심사를 분리하면 각 문서를 독립적으로 갱신할 수 있고, 목적에 따라 필요한 문서만 골라 읽을 수 있다.
